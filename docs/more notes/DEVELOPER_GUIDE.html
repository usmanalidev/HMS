1. Project overview

What is iPAC?

iPAC (Insurance Pricing and Administration for Corporate) is a Bupa Saudi Arabia internal system used to:

  • Load corporate health insurance opportunities from Microsoft Dynamics CRM
  • Model claims experience, projections, and target loss ratios
  • Apply pricing factors (cost curves, deductibles, nationality, MOH, mega-cases, seasonality, etc.)
  • Calculate premiums with authority-governed discounts and approvals
  • Post finalized rates back to CRM
  • Produce technical result analytics for renewals (member movement, hospital utilization, GQL claims, etc.)

Who uses it?

Persona Typical actions
Corporate pricer / underwriter Edit claims, target LR, save, submit for approval, post premium
Manager / higher authority Approve or reject submissions above subordinate authority
iPAC administrator Upload and maintain factor tables via Admin area
Reports administrator Access reporting screens in Admin
CRM / integration Callbacks and RPA endpoints

Business problem solved

Before iPAC, corporate pricing relied on disconnected spreadsheets and manual CRM updates. iPAC centralizes:

  • Consistent actuarial/pricing math
  • Audit trail of changes
  • Role-based authority limits
  • CRM as the commercial system of record while iPAC owns pricing detail

Technology summary

Layer Technology
UI Angular 18, Angular Material, ng2-charts, SweetAlert2
Host ASP.NET MVC 5, Razor views, .NET Framework 4.7.2
DI Autofac
Data Dapper, SQL Server stored procedures
CRM WCF net.tcp (MemberManagementRequestProcessorService)
Logging log4net, Elasticsearch (CRM integration)

2. High-level architecture

Frontend / backend relationship

iPAC is not a standalone Angular SPA in production:

  1. User opens a link from CRM → hits Home/Index?id={encryptedToken}
  2. MVC authenticates the Windows user, decrypts the token (IBM API), renders Views/Home/Index.cshtml
  3. Razor outputs <app-root OpportunityUniqueNo="..." ...>
  4. Angular bootstraps and calls MVC actions via HttpClient using environment.BaseUrl
  5. Built Angular assets live under Bupa.Ipac.Web/Scripts/ClientApp

Base href: /ngx — routes are effectively /ngx/opportunity, /ngx/projection, etc.

5. Authentication and authorization

Simple explanation

Users do not log in through an iPAC login form. Windows Authentication identifies them. iPAC checks their AD department against an allow-list, then loads their CRM profile and roles, and stores that in a server session. Angular reuses the session cookie on every API call.

Technical flow

Authorization layers

Layer Implementation File
IIS <authentication mode="Windows" /> Web.config
Department gate AuthorizedDepartments table HomeController.GetAuthenticatedUser
CRM user Retrieve.GetUserFromCRM Bupa.Ipac.CRM.Integration
MVC actions [Authorize] Business controllers
Session check [SessionExpireFilter] Returns _Logon_ JSON for AJAX
Admin [AuthorizeiPACAdmin] SessionManager.IPUser.IsAdmin
Reports admin [AuthorizeiPACReports] IsReportsAdmin
Anonymous [AllowAnonymous] CRMController.PostCRMOpportunityStatus

Session expiry (SessionExpireFilterAttribute)

  public class SessionExpireFilterAttribute : ActionFilterAttribute
  {
    public override void OnActionExecuting(ActionExecutingContext filterContext)
    {
      if (SessionManager.GetCurrentSession().IsNull())
      {
        if (filterContext.HttpContext.Request.IsAjaxRequest())
        {
          filterContext.Result = new JsonResult { Data = "_Logon_" };
        }
        else
        {
          filterContext.Result = new RedirectToRouteResult(... Account/LogOff ...);
        }
      }

Angular handles HTTP 404 in ipac.session.info.ts with a “Session Timeout” SweetAlert — align network traces with session loss.

CRM deep-link security

CRM passes an encrypted opportunity id. HomeController calls IIBMAPIService.DecryptURL(id) before loading data. Invalid decrypt → Error/InvalidRequest or Error/ServiceError.

What iPAC does not use (frontend)

  • No JWT bearer tokens in Angular
  • No CanActivate route guards
  • No HTTP_INTERCEPTORS for auth (DI hook exists but no interceptor classes)

State management

There is no NgRx. Client state is centralized in:

shared/ipac.session.info.ts (iPacSessionInfo)

  • Holds: Opportunity, IPUser, authority levels, premium rates, feature flags (MOH, delegate post, pricing adequacy)
  • Exposes BehaviorSubject streams: currentOpportunity, displaySpinner, getCostChangeConfig(), etc.
  • All HTTP via POST(url, data) → environment.BaseUrl + url

Feature services hold tab-specific dirty flags and cached data:

  • ClaimDataService — claim experience/projection
  • TargetLRDataService — target loss ratio changes
  • OpportunityService — aggregates child services for save/post/submit

HTTP pattern

// Typical call
this.session.POST("Opportunity/SaveOpportunity", payload)

// Lazy-loaded tabs (no global spinner)
this.session.POSTLazy("TechnicalResultReport/GetHospital", params)

Repository pattern

  • Base: DPGenericRepository<T>, DPTransactionRepository<T> from Bupa.Data.Dapper
  • UnitOfWork exposes 50+ repository properties (see IUnitOfWork.cs)
  • Typical method: _connection.Query<T>("sp_Name", params, commandType: StoredProcedure)

Typical new business flow

Renewal-specific concepts

  • IsRenewal flag on opportunity
  • Technical result linked via TechnicalResultID
  • Renewal authority matrix, scoring, MOH impact
  • PostPremiumRenewal vs PostPremiumNew
  • Credit control balance check (GetCreditControlBalance)

Authority and delegate post

  1. Each user has an authority level from CRM-mapped role
  2. Price increases / discounts compared against NetworkAuthorityMatrix, OpportunityAuthorityMatrix, GrossPremiumAuthorityMartix
  3. If change exceeds authority → SubmitForApproval + emails
  4. Delegate post — user posts on behalf of higher authority with audit (AuditTypes.DelegatePost)
  5. Authority post request — formal request/approval chain (AuthorityPostRequestService)

CRM integration class: Retrieve

Key operations (WCF):

  • GetOpportunityDetailsFromCrm / GetOpportunityOptionsFromCrmService
  • GetUserFromCRM
  • PostPremiumGroup — write rates back
  • GetGWPApprover
  • RetrievePremiumForIPacCorporateRequest

CRM callback

CRMController.PostCRMOpportunityStatus — [AllowAnonymous] — CRM notifies iPAC of status changes (sp_InsertOrUpdateCRMOpportunityStatus).

Notifications

Email via sp_InsertEmailNotifications and dedicated SendEmail* controller actions for approvals, delegate post, credit balance, authority post.

Audit trail

AuditTypes enum: Creation, Updation, Save, Submit, Approve, Reject, PostPremium, DelegatePost, Open, etc.
Logged through CommonClass.CreateAuditLog and AuditService.

12. Premium calculation (InRule)

Project: Bupa.Ipac.InRule
Class: InRuleEngine (e.g. CalculuatePremium)

Despite the name, this is in-process C# calculation logic, not an external InRule rules server at runtime.

Invocation path:

  1. UI → Opportunity/CalaculatePremium
  2. OpportunityService loads factors from DB (cost curves, deductibles, credibility, etc.)
  3. InRuleEngine applies formulas
  4. Results returned as JSON → session updated → premium summary UI

Admin-maintained factors (uploaded via Admin area) directly affect calculation output.

17. Debugging guide

Tools

  • Visual Studio — backend breakpoints, attach to IIS Express
  • Browser DevTools — Network tab for POST .../Opportunity/...
  • SQL Profiler / SSMS — trace stored procedure calls
  • log4net logs — application errors
  • Elasticsearch — CRM integration logs (if configured)

Breakpoint cheat sheet

Scenario Frontend Backend
App won't load app.component.ts ~line 50 HomeController.Index, GetAuthenticatedUser

Premium wrong Before save payload CalaculatePremium, InRuleEngine
Post CRM fails Post handler PostPremiumNew / PostPremiumRenewal, PostPremiumGroup

Inspect HTTP payload

  • ClaimExprince, ClaimProjection, claimData
  • targetLR, costChange
  • delegateExgratia, pricingAdequacy
  • premiumRatesWithoutCommission
  • authorityLevel, StatusReason

Log correlation

Redirect to UnAuthorized AD department not in AuthorizedDepartments DB config + AD attributes

Invalid Request page IBM decrypt failed CRM link, IBM gateway config

Known code quirks

  1. app-routing.module.ts is unused
  2. TechnicalResultModule imported eagerly and lazy-loaded
  3. debugger statement in POSTLazy (ipac.session.info.ts) — remove before production
  4. GetActiveOpportunity() uses ListOpportunity[0] in one branch — verify when multiple opportunities exist
  5. Mixed controller namespaces: Bupa.Ipac.Web.Controllers vs ipac.web.integration.Controllers
  6. Legacy common/base.service.ts hardcodes localhost URL — prefer iPacSessionInfo
